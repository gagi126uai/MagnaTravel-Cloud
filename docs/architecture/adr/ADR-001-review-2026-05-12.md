## Bloqueantes (B1..B4)

Los 4 bloqueantes deben resolverse **antes de arrancar F3.1+F3.2** del plan tactico de invariantes (`.claude/agent-memory/software-architect/project_invariantes_phase3_plan.md`).

### B1 — Falta "Migration plan" formal en el ADR

El ADR-001 describe migracion en 5 sub-fases (seccion 5.1) pero no aborda explicitamente:

- **Coexistencia `MutationGuards` vs `InvariantEvaluator`** durante el periodo de migracion. ¿Doble check? ¿Feature flag por entidad? ¿Cual gana si difieren?
- **Backfill auditor**: identificar registros pre-migracion que violan invariantes futuras (Customer.TaxId vacios, Invoice sin Letra, NC con `RefundChoice = null`). Sin auditoria, deploy de invariante con datos sucios = 500 silencioso.
- **Feature flags por invariante**: el ADR menciona `[BusinessInvariant(disabled: true)]` como rollback (seccion 5.2) pero es atributo de compilacion. Para hotfix de prod hace falta flag **runtime** (settings + admin UI).

**Remediacion**: Seccion 5 del ADR debe agregar:

1. Tabla de equivalencias `MutationGuards → IBusinessInvariant` con orden de migracion.
2. Politica de doble-check durante migracion (recomendado: feature flag por invariante con default ON en dev, OFF en prod hasta validacion).
3. Script de auditoria por invariante que detecta registros incompatibles (a correr antes del deploy de cada invariante).
4. Modelo `RuntimeInvariantFlag` (Settings table) con admin UI minima.

### B2 — NCs historicas con `RefundChoice = null` en `GetCashSummaryAsync`

El ADR introduce `Invoice.NCRefundChoice` nullable (F3.2.1 del plan). Hoy hay NCs historicas que no tienen este campo. Pregunta abierta:

- ¿`null` se trata como `EffectiveCash` (deuda contable: aparecen en CashOut retroactivamente) o como `CustomerBalance` (deuda contable: NO aparecen en CashOut pero clientes con saldos no reflejados)?
- ¿Backfill manual o automatico? ¿Por que criterio (fecha, monto, indicador en `Customer.Notes`)?

Esta decision **requiere intervencion del contador** porque cambia el Libro de Caja retroactivo. No puede tomarse en codigo.

**Remediacion**: ADR-001 debe documentar la decision tomada por Gaston + contador y explicitarla en `Invoice.IsLegacyNCMode = true | false`. `TreasuryService.GetCashSummaryAsync` debe filtrar por este flag con comportamiento explicito documentado.

### B3 — INV-066 (`NCRefundChoiceRequired`) NO es invariante de dominio

El ADR clasifica `INV-066-NCRefundChoiceRequired` como invariante con `priority: High, admitsOverride: false` (seccion 2.2, F3.2.4 del plan). Pero la regla "aprobar NC sin `refundChoice` rechaza con 400" es **validacion de input del request**, no invariante del dominio. No depende de estado de la BD, no requiere queries, no necesita el evaluator.

Si se mezclan los dos conceptos:

- La validacion de input corre dos veces (FluentValidation/DataAnnotations en controller + invariante en service).
- El catalogo de InvariantIds se contamina con reglas que no son del negocio.
- El audit log se llena de "InvariantViolation" cuando es 400 estandar.

**Remediacion**: Recategorizar INV-066 como **validacion de input** en el `ApproveCreditNoteRequest` (FluentValidation o DataAnnotation). El **verdadero invariante** asociado es:

- `INV-066-NCRefundChoiceConsistency`: si NC se aprueba con `RefundChoice = EffectiveCash`, debe existir saldo en caja suficiente (o el override correspondiente). Esto si depende de estado.

Distinguir los dos en el catalogo.

### B4 — `ApprovalRequestType.InvariantOverride` vs `FrozenEntityMutation = 5`

El ADR-001 propone usar `ApprovalRequestType.FrozenEntityMutation = 5` (existente del workflow Fase B') para el override de invariantes (seccion 2.5). Problema:

- `FrozenEntityMutation` semanticamente significa "Admin quiere editar entidad congelada por proceso operativo" (ej: cerrar reserva con saldo, modificar Customer con CAE asociado). No incluye semantica de "regla de negocio inviolable".
- Si el `ApprovalRequest.Payload` tiene que cargar `InvariantId`, el consumer (`InvariantEvaluator`) tiene que adivinar si el approval era para esa invariante especifica o para una mutacion generica de entidad congelada. Riesgo de approval valido consumido por otra invariante.

**Remediacion**: Agregar `ApprovalRequestType.InvariantOverride = 6` con payload tipado `{ InvariantId, EntityType, EntityId, OperationRequested }`. Mantener `FrozenEntityMutation = 5` para su uso original. El evaluator filtra solo approvals tipo 6 al evaluar override.

## Mejoras mayores (R1..R6)

Condiciones para subir el ADR a `Accepted`:

### R1 — Baseline p95 antes de F3.3

F3.3 migra 16 invariantes ALTA al evaluator. Cada mutacion de Payment/Invoice/Reserva corre N invariantes. **Antes de mergear F3.3**, capturar p95 actual de los 5 endpoints mas calientes:

- `POST /api/reservas/{id}/services`
- `POST /api/reservas/{id}/payments`
- `POST /api/invoices/{id}/approve`
- `POST /api/invoices/{id}/annul`
- `PATCH /api/reservas/{id}`

Target post-F3.3: incremento maximo del 15% sobre baseline. Si se excede, optimizar bucketing o cache antes de cerrar F3.3.

### R2 — Flag runtime para deshabilitar invariantes

(Esta mejora se solapa con B1 pero merece linea propia.)

El atributo `[BusinessInvariant(disabled: true)]` es **compile-time**: no sirve para hotfix de prod. Implementar:

- Tabla `InvariantRuntimeStates(InvariantId, Enabled, DisabledReason, DisabledBy, DisabledAt, ExpiresAt)`.
- Admin UI: `/admin/invariants` listado con toggle + razon obligatoria + expiracion automatica (max 7 dias).
- Audit log especial al deshabilitar/reactivar.

### R3 — Tests TDD desde F3.1, no F3.6

El plan original concentra tests en F3.6 (snapshot tests, regression suite). Esto es **anti-TDD**: las invariantes son codigo de seguridad, deben escribirse con test primero. Refactor del plan:

- **F3.1**: tests del evaluator antes de cualquier invariante real.
- **F3.2**: tests unit + integration de cada invariante de bug central NC + caja.
- **F3.3**: tests TDD por invariante migrada.
- **F3.4**: idem.
- **F3.6**: solo regression + smoke E2E + performance.

### R4 — Re-estimar F3.3+ al doble

Estimacion original F3.3: 16-24h. Realistico: **30-45h por sub-fase** considerando:

- Tests TDD por invariante (R3).
- Backfill auditor por invariante (B1).
- Re-confirmacion con expertos para invariantes ambiguas (project_invariantes_open_questions.md tiene 21 preguntas abiertas).
- Modal override + UI mensajes en frontend (F3.5 paralelo).

Effort total Fase 3 realista: **120-160h**, no 70-97h. Equivalente a 6-10 sesiones de 4-6h o 3-5 semanas calendario.

### R5 — Granularidad reserva-level vs servicio-level

Las invariantes hoy se aplican a entidades top-level (Reserva, Invoice, Payment). Pero MagnaTravel tiene **servicios por reserva** (Hotel, Vuelo, Paquete, Traslado, Asistencia). Casos:

- Una reserva con 2 hoteles, uno cancelado, uno activo. ¿Las invariantes son por reserva o por servicio?
- Voucher Issued por servicio, no por reserva. INV-036 (snapshot inmutable) aplica a servicio.
- Comision por servicio. INV-028 (`CommissionLedger.Reversed`) aplica a servicio.

**Documentar en el ADR** que el evaluator soporta ambos niveles (`TargetEntity = Reserva | ReservaService | Invoice | Payment | ...`). La deuda fiscal de reserva-level se documenta explicitamente con casos: que invariantes son por reserva, que por servicio, que mixtas.

### R6 — Refund parcial / cuotas / tarjeta no modelado

ADR-001 modela refund completo de NC (cash o balance). Casos reales no cubiertos:

- **Refund parcial**: cliente recibe 60% en cash + 40% en saldo a favor.
- **Cuotas**: NC se devuelve en N pagos parciales a lo largo de M dias.
- **Refund a tarjeta**: la pasarela rechaza, retry, time-out. Estado intermedio.

El ADR debe declarar explicitamente si entra en scope o se difiere a ADR-002 (cancelacion/refund). **Recomendado: diferir a ADR-002** porque cambia el modelo de datos (`ClientCreditEntry` con N retiros).
